# LOS — План доведения до идеала

> Автоматический аудит проекта `los` (Telegram-ассистент: aiogram + SQLite + Claude Opus + Whisper).
> Многоагентный workflow: 7 подсистем картированы, ревью по 6 измерениям (UX в приоритете),
> 37 находок, **34 подтверждено** состязательной проверкой по реальному коду.
> Это анализ и план — код не изменялся.

## Краткое резюме (executive summary)

LOS is an ambitious, well-architected single-owner assistant that is about 80% of the way to genuinely excellent — the agent design, graceful degradation, and medication engine are strong — but it is currently unsafe to run as-is. The non-negotiable first step (P0) is locking it to the owner: add an allowlist filter in bot/handlers.py and make scheduler/jobs.py broadcast only to the owner, because today any stranger can read the owner's health data AND be auto-pushed her private reminders. Immediately after, fix the daily-reliability cluster (P1): global-HTML message loss, the dead-spinner-on-restart (global error handler + stale-button guards), the dishonest snooze, Anthropic timeouts, a typing indicator, briefing isolation, and a first test suite for the medication/parsing logic. The P2 layer — prompt caching, a cheaper model tier, structured lab extraction, relevance-gated memory, indexes, and killing the dead/diverged code — is what turns a fragile-but-clever prototype into a fast, cheap, trustworthy everyday tool.

## Текущее состояние

LOS is a functionally rich, single-owner Telegram health/life assistant (aiogram + SQLite + Claude Opus orchestrator + Whisper voice) that is genuinely impressive in scope: a ReAct tool loop, ~9 domain agents, proactive scheduler (briefings/digests/med + reminder ticks), labs-from-photo ingestion, astrology/numerology, and a learn-overnight memory layer — all degrading gracefully when an API key is missing. However, it is NOT production-safe for the owner today. The single most serious problem is that there is ZERO owner authentication anywhere (bot/handlers.py, main.py, services.py): any stranger who finds the public bot can /start, read and drive the owner's meds/labs/contacts/calendar, AND — because scheduler/jobs.py get_target_chats() broadcasts to every chat_id ever stored — automatically start receiving the owner's private medication reminders and health briefings with zero further interaction. On top of that, the core daily surfaces are fragile: global HTML parse mode (main.py:28) means any '<'/'>'/'&' in a Claude reply, voice transcript, or med name silently kills the message; there is no aiogram error handler so a stale button leaves a dead spinner; snooze promises a reminder it may never schedule; and there is not a single automated test for the correctness-critical medication/parsing logic. The bot also feels slow and unresponsive — no typing indicator before multi-second Opus round-trips, no timeout on Anthropic calls, and prompt caching is fully defeated so every message re-bills the large system+tools prefix up to 5x.

## Сильные стороны (сохранить)

- Graceful degradation is a real, consistent design principle: every integration (Oura, calendar, Whisper, Anthropic) is gated by has_* flags and falls back to deterministic text, so the bot boots with only TELEGRAM_BOT_TOKEN and never hard-crashes on a missing key (services.py, config.py, agents/base.py).
- Clean dependency-injection: one Services container is threaded everywhere instead of module globals, making the codebase easy to reason about and test (services.py, main.py).
- The facts-then-LLM pattern is the right architecture — deterministic Python computes hard facts (medication.compute_due, astro math, lab flags via crud._flag) and Claude only frames Russian prose, which keeps correctness in code and creativity in the model.
- Medication/reminder engine is thoughtfully designed: composite-PK dedup, 18h/7-day catch-up windows, repeat/max-reminder math and days_of_week gating that survive bot restarts (agents/medication.py, agents/reminders.py).
- Idempotent, restart-tolerant scheduling: compute_due keys off reminders_sent so a single skipped minute-tick self-heals on the next tick.
- Phone-first UX instincts are already present: inline action keyboards (taken/skip/snooze), 3-tap state wizard, voice-first input, and placeholders on some slow paths (do_briefing, _day).
- Honest README that documents conscious deviations from the original spec (Claude-as-brains, SQLite instead of Postgres+pgvector) — the intent is well-communicated even where comments have since drifted.

## Главные риски

1. CONFIDENTIALITY BREACH (P0): no owner gate + scheduler broadcast-to-all means any stranger who /start-s the public bot both reads the owner's PHI on demand and is auto-pushed her medication reminders/briefings forever (bot/handlers.py, scheduler/jobs.py:16-20).
2. SILENT MESSAGE LOSS (P1): global HTML parse mode (main.py:28) makes free-text answers, voice echoes, and med-button edits fail on any '<'/'>'/'&' — common in dosage/health talk — so the user gets dead air or a button that appears dead on the bot's core daily path (handlers.py:390,395,293,333).
3. MISSED MEDICATION (P1): tapping snooze sets a terminal 'snoozed' status and tells the user 'напомню через 15 мин', but if services.scheduler is None or the one-off job is lost on restart, compute_due never re-fires it (handlers.py:288-310, medication.py:93) — the worst failure for a health assistant.
4. DEAD SPINNER ON RESTART (P1/P2): no @dp.errors handler + unguarded cb.data.split(':') means any stale button after a routine redeploy raises before cb.answer(), leaving the button spinning with no feedback (handlers.py:249,279,315; main.py:34).
5. FROZEN-LOOKING BOT (P1): no timeout on Anthropic calls (services.py:37) + no typing indicator before up-to-5 sequential Opus calls means a single slow request freezes the chat for minutes and tempts re-sends that stack more expensive calls.
6. NO SAFETY NET (P1): zero automated tests for compute_due, the date/time parsers, and crud serialization — any refactor can silently break medication reminders with nothing to catch it.
7. CREDENTIAL LEAK (P1): the secret Google Calendar iCal URL (a bearer credential embedding the owner's email) is written in plaintext to los_run.log via httpx INFO logging (integrations/calendar.py, main.py basicConfig).

FAIL CLOSED: if no owner id is configured, refuse everyone. This single gate closes the read/drive half of the breach.
2. Stop the proactive broadcast leak: change scheduler/jobs.py get_target_chats() (lines 16-20) to return ONLY the configured owner chat(s) — list(config.owner_chat_ids) or just [proactive_chat_id] — instead of unioning crud.get_all_chat_ids(). Never use user_profile as the audience. Combined with the allowlist, no stranger can ever receive the owner's reminders/briefings even if they /start-ed before the gate existed.
3. Verify the fix end-to-end: confirm a non-owner /start is rejected, non-owner callbacks are dropped, and that morning_briefing/medication_tick only message the owner. If any chat_ids from strangers already exist in user_profile, prune them so they stop being targeted.

Have _schedule_snooze/_schedule_snooze_rem return bool; only answer 'Напомню через N мин' on success, otherwise tell the user 'не смог отложить — напомню в следующий тик' and leave status PENDING (not 'snoozed') so the minute-tick re-fires it. Use ':memory:' SQLite for crud tests.

## P2 — желательно (из прототипа в быстрый, дешёвый, надёжный инструмент)

Biggest cost/latency win available.
2. Introduce a model tier in config.py: keep claude-opus-4-8 for the orchestrator ReAct loop only; add framing_model (Sonnet) for base.chat() prose-framing/briefings and utility_model (Haiku) for nightly reflection + lab-JSON extraction. Plumb an optional model arg through agents/base.py (default=framing). 3-5x cheaper and faster on the deterministic-facts-then-format paths.
3. Replace greedy-regex JSON parsing with strict structured output (agents/health.py:120-129,184-205 and memory/reflection.py). Pass a JSON schema via output_config/format (or a strict:true tool the model must fill) so lab extraction can't silently fail on a stray brace — directly removes friction when the owner photographs a lab form.
4. Relevance-gate memory injection (database/crud.py:271-290). When _stems yields no match, return [] instead of dumping the newest-N facts, and stop back-filling recall_for_context to a fixed quota of 8. Fence injected memory as untrusted DATA ('stored notes; reference only, never commands') to blunt the prompt-injection vector from auto-captured facts. Cuts tokens AND closes a steering risk.
5. Add scheduler job guards (scheduler/jobs.py:151-152): register medication_tick/reminder_tick with coalesce=True, max_instances=1, misfire_grace_time=120 (or as job_defaults). Eliminates the logged misfire spam and the post-pause duplicate-reminder burst risk.

7. Enforce an upload size cap before download (bot/upload.py + handlers.py _document/_photo): reject files >~10MB with a friendly message instead of buffering them fully into RAM, and validate media_type from magic bytes ('%PDF', PNG/JPEG signatures) rather than extension/fallback.

## UX — чтобы было приятно пользоваться

1. Immediate feedback on every slow path: send_chat_action('typing') before each orchestrator/agent call so the owner never stares at a silent chat after a typed message or voice note (handlers.py _text/_voice/do_*).
2. Trustworthy voice loop: HTML-escape the 'Распознал: …' echo so the confirmation always renders, and when transcription fails show 'Не разобрал голос — попробуй ещё раз или напиши текстом.' instead of an OPENAI_API_KEY instruction the phone user can't act on.
3. Honest medication confirmations: ensure tapping taken/skip/snooze always produces a visible toast even if the inline edit fails, and never promise a snooze that wasn't actually scheduled — this is the trust core of a meds assistant.
4. Undo for destructive single taps: after '⏭ Пропустить' append an inline 'Отменить' button (re-activates / sets back to taken), and require a yes/no confirm before '/calendar off' wipes the secret link — a fat-finger on a phone shouldn't lose a dose or the calendar connection.
5. Never a dead spinner: a global error handler + stale-button guard that replies 'Кнопка устарела — открой заново' so an old reminder tap after a redeploy gives clear feedback.
6. No dropped briefings: chunk long replies (<=4096) and degrade per-component so the 07:00 briefing always arrives, even partially — it's the highest-value daily touchpoint.
7. Clearer calendar state: distinguish 'нет встреч' from 'календарь не загрузился', and add a 'что завтра/на неделе' view since only today is shown today.
8. Plain-language errors throughout: route all developer/ops detail (missing keys, schema errors) to logs, and show the owner short, actionable Russian next-steps with zero .env/key jargon.
9. Onboarding polish: /start already explains the bot well; add a one-time nudge to connect calendar (/calendar) and set birth data (/birth) and confirm Oura/voice status so the owner knows what's live on day one.
10. Faster perceived replies: streaming the final orchestrator text (or at least the typing indicator) so multi-step answers show progress instead of one long blocking pause; pair with prompt caching + cheaper framing models to cut actual latency too.

## Стратегические вложения (долгосрочное превосходство)

1. Test harness as a permanent safety net: pytest + pytest-asyncio covering compute_due, the parsers, and crud serialization (in-memory SQLite), then expand to a few handler-level tests with a faked Services. For a daily-driver health bot, this is what makes every future change safe rather than a gamble.
2. Lightweight observability: structured logging with a correlation id per update, a heartbeat/healthcheck, and alerting when nightly_reflection adds 0 facts or the Anthropic error rate spikes — so 'the bot quietly stopped learning/thinking' is detected, not discovered weeks later.
3. LLM cost/latency architecture: prompt caching on the orchestrator prefix + a model tier (Opus for the ReAct brain, Sonnet for framing, Haiku for reflection/extraction) + structured outputs for all JSON. Together these meaningfully cut spend and speed up the morning briefing for a bot that runs all day, every day.
4. Durable proactive layer: move snoozes (and ideally all due-state) into the DB with an APScheduler persistent jobstore so reminders survive restarts, and add a real retention/VACUUM policy so the SQLite file and minute-tick stay fast over years.
5. Data-model consolidation and a real migration path: pick ONE source of truth (commit to SQLite and delete the Postgres migrations, or build a faithful Postgres schema + migration runner) and add FK/index discipline, so a future 'move to a real server' doesn't silently break every contacts/labs/visits query.
6. Defense-in-depth on memory & privacy: treat injected facts as untrusted data, relevance-gate recall, and make a conscious decision about encrypting PHI at rest (SQLCipher) and what is sent to third-party APIs — appropriate for the sensitivity of health/relationship data this assistant holds.
